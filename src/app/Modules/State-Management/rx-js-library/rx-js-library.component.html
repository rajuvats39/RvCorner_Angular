Subject VS BehaviourSubject

Both Subject and BehaviorSubject are Observables

Subject

A Subject is a special type of observable that allows values to be multicasted to many observables. In simple terms, they are special containers that can hold values, and when a new value comes into a subject, that value can be shared with numerous observers at once.

A Subject is a special type of Observable that allows values to be multicast to multiple observers. Unlike regular observables, a Subject can both emit values (as an Observable) and be subscribed to (like an Observer).

Key Characteristics:

No Initial Value:Subject does not store any current value. When a new subscriber subscribes, it does not receive any previously emitted value.

Multicasting: It allows multiple observers to listen to the same subject.

Manually Emitting Values: It allows you to emit values manually using next(), It means whenever we requird the subscription on action needed.

It is used whenever we requird emiting and subscribing data with action subscription

BehaviorSubject

A BehaviorSubject is a type of Subject that requires an initial value and always provides the current value to any new subscribers. It stores the last emitted value, so when a new observer subscribes, it immediately receives the most recent value.

Key Characteristics:

No Initial Value:A BehaviorSubject must be initialized with a default value.

Stores the Latest Value: It stores the last emitted value and provides it to new subscribers even if they subscribe after the value has been emitted.

Multicasting: Similar to Subject, it allows multiple subscribers.

It is used whenever we requird continious emiting and subscribing data without any action subscription

When to Use:

Subject:Use it when you do not need to remember the last emitted value. Useful for things like button clicks, where you want to notify subscribers only about the actions that occur after they subscribe.

BehaviorSubject:Use it when you need to remember the last emitted value and ensure that new subscribers receive that value immediately. Ideal for things like form states or data models where you want to retain and share the latest state across different parts of your application.
